25.02.10 Explaining Deep Learning-based Network Intrusion Detection Systems for Active Intrusion Responses

연구 배경 및 동기

  • 최근 딥러닝 기반 NIDS가 많은 연구에서 우수한 성능을 보인다고 함
  • NIDS의 기본 원리는 네트워크 트래픽을 모델에 학습시켜 점수를 부여, 이를 기반으로 악성 여부를 분류하는 것임
    • 기존의 탐지 시스템은 크게 두가지로 나뉨
      • 시그니처 기반: 기존 패턴과 비교를 통한 탐지
      • 통계기반: 이상 행위 분석을 통한 탐지
  • 기존 모델과 NIDS의 비교
    • 장점
      • 미탐지(Zero-day attack) 탐지 가능
      • 복잡한 패턴 탐지에 뛰어남
    • 단점
      • 탐지 결과의 해석 가능성 부족
      • 오류 발생시 높은 비용
        • FN(False Negative, 탐지 실패):
        • FP(False Positive, 오탐):

기존 연구의 한계점

  1. 탐지 결과와 실행 가능한 해석 간의 의미적인 차이(semanti gap)이 존재
    • 네트워크 운영자가 탐지된 이상을 바탕으로 즉각적인 방어 조치에 어려움 발생함
  1. 탐지 오류(FP, FN) 발생시 가능
    • 오탐의 경우 → 운영자는 오탐을 막기 위해 코스트를 소모하게 됨
    • 미탐의 경우 → 보안 위협 초래
  1. 능동적인 방어 조치 불가능
    • 기존 DL-NIDS는 탐지된 위협에 대해 수동적인 경고(Alert)만 제공하고, 실질적인 대응(방어 규칙 적용)은 사람이 해야 함
    • 설명 기법을 적용하면 탐지 결과를 분석하여, 자동으로 방어 규칙을 생성하고 네트워크 방어 시스템에 적용할 수 있음

⇒ 따라서 DLNIDS 가 탐지 하는 과정에 대한 설명 가능성이 중요함

XNIDS의 필요성

기존의 방법들은 특정 입력 샘플과 그에 대한 예측 결과를 해석하는 국소적 설명을 제공하는데 집중함

→ ex) Gradients, Integrated Gradients(IG), Layer-wise Relevance Propagation(LRP), LIME 등

하지만 위 기존 설명 방법들은 효과적이지 못함

  1. DL-NIDS의 입력 데이터는 시간 종속적(time-dependent)임
    • 기존 설명 기법들은 대부분 단일 입력 샘플에 대한 해석만 수행
      방법문제점
      Gradients(IG, LRP)현재 입력만을 고려하여 과거 트래픽 패턴을 반영하지 못함
      LIME, SHAP개별 샘플에 대한 지역적(local) 설명만 제공하여 시간적 흐름을 반영할 수 없음
      LEMNA보안 애플리케이션에 특화된 방법이지만, 시계열 특성을 반영하지 못함
  1. 특징 의존성(Feature Dependency)을 반영하지 못함
    • 네트워크 트래픽 데이터는 구조적인 특징(feature dependency) 을 가지고 있으나 모든 특징을 독립적인 개별 요소로 가정하기 때문에 관계를 반영하지 못함
    • ex)
      방법문제점
      Gradients(IG, LRP)각 뉴런의 기여도를 개별적으로 분석하지만, 특징 간의 구조적 종속성을 고려하지 않음
      LIME, SHAP각 특징(feature)의 중요도를 개별적으로 계산하므로 특징 간의 관계를 반영하지 못함

  1. 방어 규칙(Defense Rules) 생성의 어려움
    • 정확성(Precision) vs. 일반화(Generalization)
      • 너무 구체적인 규칙 → 과적합(Overfitting) 발생, 관리 부담 증가
      • 너무 일반적인 규칙 → 정상 트래픽까지 차단하는 부작용
    • 다양한 방어 시스템과의 호환성
      • OpenFlow, IP Table 등 각 방어 시스템마다 규칙 형식이 다름 → 범용적인(Unified) 규칙이 필요

    → XNIDS는 정확한 방어 규칙을 생성하는 것을 목표로 함

연구 목표: XNIDS(NIDS 설명 모델)

해결해야 하는 목표

  1. 시간적인 연관성 반영
  1. 트래픽 간의 특징 의존성 반영
  1. 실용적인 방어 규칙 생성을 위해 정밀성과 일반화 사이의 균형
  1. 다양한 방어 도구에 사용 가능한 보편적인 방어 규칙 생성

핵심 기법

  1. 최종 목표
    • DL-NIDS의 탐지 결과를 최대한 정확하게 설명(fidelity 유지)하면서도,
    • 불필요한 특징을 최소화(sparsity 적용)
    • 탐지된 패킷이 과거 입력의 영향을 받았을 때, 최소한의 과거 입력만 사용하도록 선택

1. 히스토리 입력 근사 (Approximating History Inputs)

  • DL-NIDS는 단일 입력만 분석하는 것이 아니라 과거의 네트워크 트래픽까지 고려해야 함
  • XNIDS는 탐지 결과에 영향을 미치는 중요한 과거 입력을 자동으로 선택
  • 사용된 수식
    • f는 DL-NIDS이며 오차율이 임의의 값보다 작은 가장 작은 l을 찾는게 목표임
    • x_t는 현재 탐지된 패킷을 의미(1*d(속성))
  • 불필요한 입력 제거
    • 호스트 필터: 같은 호스트에서 발생한 입력을 제거합니다.
    • 프로토콜 필터: 동일한 프로토콜을 가진 입력을 제거합니다.
  • 최종적으로 X’_t,m 이라는 m*d의 행렬 생성됨

2. 패킷 가중치 랜덤 샘플링

  • 목표 패킷에서 시간적으로 멀수록 해당 패킷의 특징을 확률적으로 0으로 제거

    → 특정 특징(시간 지연)만 제거하여 데이터에서 중요한 요소를 강조"는 과정

  • 사용된 수식
  • 최종적으로 m*d 행렬 생성됨

3. 특징 의존성 분석 (Feature Dependency Analysis)

  • 네트워크 패킷 데이터는 구조적 특징을 가지므로, 기존 설명 기법이 모든 특징을 개별 요소로 가정하는 것은 부적절
  • XNIDS는 특징 간의 관계를 분석하여 보다 정확한 설명을 제공
  • 사용된 수식
  • 각 패킷은 특징 1개를 제외하면 0이됨

4. 희소 그룹 라쏘 (Sparse Group Lasso, SGL)

  • 탐지 결과에 영향을 미치는 최소한의 핵심 특징만 선택하여 설명을 단순화
  • 설명을 간결하고 해석 가능하게 만듦
  • 구조적으로 연관관계를 나타내기 위해 해당 방법 사용(2번째 항)
  • 사용된 수식
    • 제 1항: 손실 최소화(Fiedelity)
      • DL-NIDS 모델 f(·)를 선형 모델 g(·)로 근사 시킴

        ⇒ XNIDS가 DL-NIDS의 탐지 결과를 최대한 정확하게 설명하도록 학습

    • 제 2항: 그룹 단위 희소성(Group Sparsity, L2 Regulariztion)
      • 특정 특징 그룹 전체를 제거할 수 있도록 유도
      • 그룹 q의 가중치 벡터 β_q의 L2 노름(벡터 크기)을 최소화
      • \sqrt{p_q}: 특징 그룹 크기에 따른 가중치 조정 (큰 그룹이 작은 그룹보다 불리해지지 않도록 함)

        ⇒ 탐지 결과에 영향이 적은 그룹은 β_q = 0이 되어 전체 그룹이 제거됨

    • 제 3항: 개별 특징 단위 희소성(Feature Sparsity, L1 Regularization)
      • 그룹 내에서도 개별 특징을 제거할 수 있도록 유도
      • β_i = 0이 되도록 강제하여 불필요한 개별 특징을 제거

        ⇒ 그룹이 유지되더라도 그 안에서 중요한 몇 개의 특징만 남게 됨

    • 정리
      • 두번째 arg min 과정을 통해 설명모델 g()를 학습 시키는 것이 최종 목적임
      • 해당 선형근사 모델을 통해 최종적으로 DL-NIDS 탐지 결과를 최대한 가깝게 예측하는 근사 모델로 사용됨
        ← 비선형 모델인 DL-NIDS는 실시간 해석이 어려움으로 탐지한 패킷을 선형 근사하여 대략적인 원인을 찾는것이 목표인것임

5. 최종 결론

  • DL-NIDS가 탐지한 탐지 결과를 목표 패킷 x_t로 두어 해당 패킷을 선형 근사를 통한 패킷의 특징을 탐지하여 원인을 도출해 내는것이 결론임

방어 규칙 생성

  • 목표:
    • DL-NIDS가 탐지한 공격 패킷을 기반으로 자동 방어 규칙을 생성
    • 다양한 방어 시스템(iptables, OpenFlow 등)에서 적용할 수 있도록 보편적인 방어 규칙(unified defense rule)으로 변환
    • 탐지된 패킷을 얼마나 강력하게 차단할지(블록 전략) 결정 가능
  • 규칙 생성 과정
    1. 방어 규칙 범위(Defense Rule Scope) 결정

      → 특정 플로우 / 특정 호스트 / 여러 호스트 차단 여부 결정

    1. 차단 전략(Block Strategy) 적용

      → 패킷 차단 or 호스트 차단 여부 결정

    1. 보편적인 방어 규칙(Unified Defense Rule) 생성

      → 방어 시스템에 적용 가능하도록 변환

1. 방어 규칙 범위 결정

범위는 3 단위로 결정됨, 플로우 단위, 호스트 단위, 다중 호스트 범위로 구분지을 수 있음

  • 플로우 단위 범위 (Per-flow Scope)
    • 특정 네트워크 플로우에서 전송되는 패킷에만 영향을 미치는 방어 규칙이다.
    • 예를 들어, 특정 TCP 연결을 종료하는 방어 규칙은 플로우 단위 방어 규칙에 해당한다.
  • 호스트 단위 범위 (Per-host Scope)
    • 동일한 호스트에서 발생하는 여러 개의 플로우를 차단할 수 있는 방어 규칙이다.
    • 예를 들어, 봇(Bot)을 차단하는 방어 규칙은 호스트 단위 방어 규칙이다.
  • 다중 호스트 범위 (Multi-hosts Scope)
    • 여러 호스트에서 발생하는 여러 개의 플로우를 차단할 수 있는 방어 규칙이다.
    • 예를 들어, 특정 집합의 호스트(예: 봇넷)에서 발생하는 SYN 요청을 차단하는 방어 규칙은 다중 호스트 범위의 방어 규칙이다.

해당 범위로 분류하기 위해 탐지결과 설명을 분석한다

  • 통계 정보(Statistical Information) S 테이블
    필드설명
    IP_pool입력에 포함된 IP 목록
    IP_n동일한 IP에서 발생한 최대 패킷 수
    MAC_n동일한 MAC 주소에서 발생한 최대 패킷 수
    Port_n동일한 포트에서 발생한 최대 패킷 수
    Protocol_n동일한 프로토콜에서 발생한 최대 패킷 수
  • 분석 방법
    1. Statistical Information (S)에서 가장 큰 값을 가지는 필드를 확인
    1. 필드를 분석하여 해당하는 방어 규칙 생성
      • Protocol_n , Port_n 값이 가장 큰경우

        ⇒ 이상 트래픽이 동일한 프로토콜을 사용하는 여러 호스트에서 발생했음을 의미.

        ⇒ 다중 호스트(Multi-hosts) 단위의 방어 규칙 적용

      • 이상 트래픽이 동일한 호스트에서 발생한 경우

        ⇒ 호스트 단위(Per-host) 또는 플로우 단위(Per-flow)의 방어 규칙 적용

      • 높은 이상치를 포함하는 값들이 많은 경우

        ⇒ 해당 호스트가 여러 프로토콜을 사용하여 공격을 수행 함

        ⇒ 호스트 단위(Per-host) 방어 규칙 적용

2. 차단 전략 적용

네트워크 환경의 변화에 적응할 수 있도록 네트워크 운영자가 설정할 수 있는 보안적 제약

  • 소극적 차단(Passive Block)
    • XNIDS는 악성 플로우(Malicious Flows) 만 차단하는 방어 규칙을 생성한다.
    • 이 전략의 목표는 방어 규칙이 네트워크 환경에 미치는 영향을 최소화하는 것이다.
  • 적극적 차단(Assertive Block)
    • 네트워크 운영자는 DL-NIDS 및 XNIDS가 생성한 방어 규칙을 신뢰하는 경향이 있다.
    • 따라서 생성된 방어 규칙을 보다 적극적으로 적용한다.
  • 공격적 차단(Aggressive Block)
    • XNIDS는 악성 활동(Malicious Activities) 을 근본적으로 차단하기 위해 악성 호스트(Hosts)를 직접 차단하는 경향을 가진다.
    • 이 전략의 목표는 악성 호스트를 네트워크 환경에서 신속하게 제거하는 것이다.

3. 보편적인 방어 규칙 생성

탐지 설명 결과와 실제 적용 가능한 방어 규칙인 통합 방어 규칙 표현을 제안

  • 통합 방어규칙 생성

    방어 규칙 범위를 방어 조치와 연결하기 위해 두 가지 기본적인 연산을 정의한다.

    • drop_flow: 플로우 단위(Per-flow) 방어 규칙에서 악성 플로우를 차단.
    • drop_host: 호스트 단위(Per-host) 방어 규칙에서 악성 호스트를 차단.
  • 차단 전략과 연계

    차단 전략에 따라 drop_flow 및 drop host 연산이 변경됨

    • 소극적 차단(Passive Block)
      • drop_host 연산을 drop_flow 연산으로 변경.
    • 공격적 차단(Aggressive Block)
      • drop_flow 연산을 drop_host 연산으로 변경.
    • 적극적 차단(Assertive Block)
      • 연산을 변경하지 않고 그대로 유지.
  • 최종 방어 규칙 생성
    • 방어 규칙을 생성할 때, 설정된 연산(Operation)에 따라 엔터티(Entity)를 생성한다.

      ex) 다중 호스트(Multi-host) 방어 규칙을 drop_flow 연산과 함께 생성할 경우, 플로우(Flow) 엔터티를 재귀적으로 생성한다.

    • 규칙 충돌 방지를 위해, 우선순위(Priority)를 증가하는 숫자로 설정한다.
    • Timeout(유효 시간)은 배포 환경에 맞게 조정 가능하다.

⇒
