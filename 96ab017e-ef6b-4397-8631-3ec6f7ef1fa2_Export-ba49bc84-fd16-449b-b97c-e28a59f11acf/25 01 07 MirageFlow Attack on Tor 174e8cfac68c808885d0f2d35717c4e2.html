25.01.07 MirageFlow Attack on Tor

01. Introduction

  • 일반적인 Tor의 공격 벡터
    • 웹사이트 핑거프린팅
    • 라우팅
    • end-to-end correlation(종 단간 상관관계)
    • congestion(혼잡)
    • side channel attacks

    ⇒ 각 공격의 공통점: 공격자는 자신의 서버나 릴레이로 최소한의 리소스로 최대한 많은 사용자의 트래픽을 유도해야함

Tor 대역폭 측정 메커니즘과 그 약점

  • 메커니즘
    • 정기적으로 네트워크의 릴레이에 대한 상태 정보를 합의 파일(consensus file) 형태로 저장, 유지, 관리함
    • 해당 목록에는 각 릴레이에서 사용 가능한 추정 대역폭이 포함됨
    • Tor 클라이언트는 가장 최근의 합의 파일을 통해 경로를 선택함
    • 이때 대역폭 측정은 릴레이를 통과하는 두 개의 릴레이로 측정 회로를 설정하고 웹 서버로 데이터를 업로드/다운로드 하는 방식으로 대역폭 측정함
  • 약점
    • 문제는 해당 측정 과정에서 2개의 릴레이만으로 구성됨
    • 익명성을 보장하려면 최소 3개의 릴레이를 통과해야함(middle, exit, bridge)
    • 시나리오
      • 공격자가 측정을 탐지 했을 때, 공격자는 모든 대역폭을 해당 측정에 몰아넣어 최대 177배의 부풀린 대역폭을 얻을 수 있음
      • 이를 통해 실제 대역폭 용량과 상관없이 거짓 정보를 제공, 사용자 트래픽 점유율을 높이려고 시도함

첫 발견: 릴레이 운영자의 무제한 리소스 공유 가능

  • 현재 Tor는 릴레이 운영자가 물리적 머신이나 네트워크 링크와 같은 공유 자원을 사용할 수 없도록 제한하지 않음
  • 공격자가 여러 릴레이를 단일 물리적 호스트의 가상머신으로 실행가능
  • 별도의 물리적 머신에 위치하더라도 동일한 네트워크 링크를 공유 가능

    ⇒ 공격자가 측정 트래픽을 탐지, 대역폭을 늘려 공격자가 제어하는 릴레이 노드에 타겟팅하여 사용하도록 만들 수 있음

  • 주요 기여(?)
    • 새로운 대역폭 인플레이션 공격 전략: 클러스터 내 릴레이 간 자원 공유를 활용하여 대역폭을 부풀리는 MirageFlow 전략을 제안 및 조사
    • 두 가지 MirageFlow 변형: C-MirageFlow(Co-resident Relays)와 D-MirageFlow(Dedicated Server)를 제안합니다.
    • 공격 평가: 소규모 Tor 테스트 네트워크에서 두 가지 공격을 평가하여 각각 최대 n과 n ∗ N의 인플레이션 계수 달성, 전체 서버 클러스터 크기가 15hop 이내면 선형적으로 효과가 증가하지만 매우 커질 시 선형 성장의 약 76%로 제한됨
    • 기존 대역폭 스캐닝 대안의 저항성 연구:
      • 최신 머신러닝(ML) 기반의 대역폭 측정 기법이 MirageFlow와 같은 공격에서 효과적으로 방어할 수 있는지 검토하였지만 기계학습을 통해 F1스코어 99%, 0.5ms로 정확히 탐지 가능한걸 확인하였음

        ⇒ 릴레이 증가가 근본적인 해결책이 되지 못함

      • 측정 트래픽을 완전히 제거하고 대역폭 추정을 위해 사용자 트래픽을 활용하는 방법만이 인플레이션에 탄력적인 대역폭 측정 솔루션을 제공할 수 있을 것으로 판단함
    • Tor 네트워크 분석:
      • 실제 네트워크에서 MirageFlow 공격이 진행중인지 분석을 실시
      • 특정 릴레이에서 해당 특성이 드러났다고 함
        • 유사한 대역폭과 가동 시간(uptime)을 가진 릴레이 패밀리

          ⇒ 특정 릴레이 그룹이 비슷한 대역폭, 가동시간을 공유하고 있음

          ⇒ 이는 해당 릴레이들이 공동으로 MirageFlow 공격을 수행함을 알 수 있음

        • 대체 포트 번호 사용

          → 일부 릴레이에서 Tor가 사용하지 않은 대체 포트 번호를 사용하고 있음이 관측됨

          ⇒ 여러 릴레이가 Co-resident실행(같은 물리적 서버에서 동시 실행)되고 있음을 시사함

      • 그러나 이러한 특징을 통해 완벽히 악의적인 공격을 확인할 수 있는 방법이 부족함
      • 이를 위해 해당 전략 제안

        ⇒ 발견한 특성(대역폭, uptime, 대체포트 사용)을 기반으로 MirageFlow 공격을 탐지

02. Background

Tor Relays and Circuits

  • Tor 유저는 최소 3개 이상의 relay를 통해 네트워크를 구성함
    • Guard relay
      • 최초 hop, 연결의 진입점 역할을 함
      • 로테이션 되기 전에 최소 2-3개월 사용됨
      • 이를 위해선 충분한 가동 시간(8~68D)과 최소 대역(2Mbps)를 충족해야함
    • exit relay
      • 익명성 보존 통신 회로의 최종 릴레이
      • 미지막 hop을 의미
      • 클라이언트의 트래픽을 Tor 네트워크를 넘어 인터넷의 목적지로 전송함
    • 최종적으로 exit relay의 ip만 보이게 되며 guard와 exit 사이를 middle relay라 함

Directory and Bandwidth Authorities

Bandwidth Measurements in Tor

최초 SBWS(Simple Bandwidth Scanner)가 대역폭 측정 시 0~1 GB를 다운, 5초 이내에 다운시 0~16 or 16~32mb를 늘려 다시 다운, 10초 이내에 응답시 해당 범위 만큼 용량을 다운 시킴으로서 대역폭 측정

Building Consensus

  • MirageFLow 종류
    • C-MirageFlow
      • 여러 개의 Tor 릴레이를 단일 서버 내에서 함께 운영하며, 측정 트래픽이 감지되면 해당 릴레이에 서버의 모든 자원을 집중시켜 대역폭 측정치를 부풀리는 방식
      • 대역폭을 최대 n배로 늘림(n: 단일 서버에서 호스팅된 클러스터 크기)
    • D-MirageFlow
      • 여러 서버에 분산된 릴레이 클러스터를 구성하고, 측정 트래픽이 감지되면 이를 강력한 전용 서버로 우회시켜 대역폭을 부풀리는 방식
      • 대역폭을 최대 n*N/2배로 늘림(N: 서버수)

    ⇒ 결과적으로 100 MB/s 대역폭의 10개의 전용서버와 109개의 릴레이 클러스터를 사용하여 Tor 네트워크의 트래픽 절반을 제어 가능해짐

  • 위험한 이유
    • Tor 네트워크의 기본 보안 모델과 사용자 프라이버시를 근본적으로 위협하기 때문
    • Tor는 인터넷 사용자의 익명성을 보장하기 위해 설계된 네트워크이지만, MirageFlow는 Tor 네트워크의 대역폭 측정 및 트래픽 분배 메커니즘의 취약점을 악용하여 공격자가 트래픽을 조작하거나 더 큰 제어권을 확보할 수 있게 만듦
    • 대역폭 측정 조작의 여파
      • Tor 네트워크는 릴레이의 대역폭을 측정하여 트래픽을 할당
      • MirageFlow는 가짜 대역폭 데이터를 만들어 공격자 릴레이가 더 많은 사용자 트래픽을 처리하도록 만듦

        ⇒공격자는 더 많은 사용자의 데이터를 처리하거나 관찰 가능

        ⇒ 더 많은 사용자의 트래픽 분석 가능

        ⇒ 공격자가 시작 및 종료 릴레이를 동시에 통제하는 경우 사용자의 활동 추적도 가능

결론 및 주관

  • 해당 취약점은 Tor의 익명성을 위협하는 취약점으로 보완책을 제안, 테스트 해보았지만 완벽하지 않음
  • Tor 관계자는 이미 해당 취약점을 인지하고 있는 듯 함(영상 뒤에 해당 내용과 관련된 정황 O)
